ASSISTANT: ## IPGU (입구 — "Entrance")

An automated **bilingual subtitle generation pipeline** that takes a video file, transcribes it with Google Gemini's multimodal API, translates it with either Gemini or Claude, and outputs a styled SRT file.

### Pipeline

```
Video → Split (ffmpeg) → Transcribe (Gemini) → Translate (Gemini/Claude) → Parse → Validate → Finalize → Bilingual SRT
```

An optional reference SRT can be fed in to guide translation quality and timing.

---

### Notable Techniques

**1. Queue-based concurrency with `Promise.race`**
Rather than `Promise.all` or `Promise.allSettled`, chunks are processed via a `startNextTask()` pattern that feeds a bounded queue — new work starts as soon as a slot opens, keeping concurrency at exactly `maxConcurrent`.

**2. Multi-level retry with progressive leniency**
Three distinct retry layers:
- **API retries** — exponential backoff for transient failures
- **Validation retries** — if the LLM output fails parsing/validation checks, the entire translate→parse→validate cycle is re-run
- **Last-chunk fallback** — on the final chunk's final attempt, validation thresholds are relaxed and the "best" prior attempt (most parsed entries) is accepted rather than failing the whole pipeline

**3. Chunk overlap resolution**
Chunks are split with configurable time overlap (e.g., 120s). During finalization, overlapping subtitle entries are deduplicated by keeping the version from the **later** chunk (which has more surrounding context). A multi-pass algorithm (up to 10 iterations) then repairs timing overlaps by shortening earlier entries while respecting min/max duration constraints (0.5s–7.0s).

**4. Relative→absolute timestamp adjustment**
Gemini transcribes each chunk with timestamps relative to chunk start (`mm:ss`). These are converted to absolute timeline positions by adding the chunk's `startTimeSeconds`, allowing the final SRT to reconstruct a continuous timeline from overlapping fragments.

**5. Dual-model architecture with unified cost tracking**
Gemini (multimodal upload + streaming) and Claude (text streaming with extended thinking) are abstracted behind a common interface. Token usage is tracked per-attempt and accumulated across retries, with per-model pricing from `config/models.ts` producing a final cost report.

**6. Template-driven LLM prompts**
Translation prompts use a template file with placeholders (`{ADJUSTED_TRANSCRIPT}`, `{REFERENCE_SRT}`, `{TARGET_LANGUAGE_TAG}`, etc.), making prompt iteration independent of code changes.

**7. Structured LLM output parsing with issue tracking**
The parser extracts `<subline>` XML blocks from LLM responses (handling both markdown-fenced and bare formats), tracks every malformed tag or missing field as a `ProcessingIssue` with severity/context, and feeds these into a multi-threshold validator (10% error rate, 80% ID coverage, 80% count match, 25% timing mismatch tolerance).

---

### Evolution from `old_code/`

The `old_code/` directory contains the original monolithic scripts (~2000 LOC each). The rewrite decomposed these into focused modules (100–600 LOC), added Commander.js presets, typed config, dual-target logging, and the full retry/validation/cost-tracking infrastructure — turning a proof-of-concept into a production-grade tool.

ASSISTANT: ## 1. Chunking is purely time-based, not content-aware

Looking at `calculateChunks` in `src/utils/time_utils.ts:85-124`:

```
totalDuration = 3600s, chunkDuration = 1200s, overlap = 120s

Chunk 1: [0s ————————— 1200s]
Chunk 2:          [1080s ————————— 2280s]
Chunk 3:                    [2160s ————————— 3360s]
Chunk 4:                              [3240s ———— 3600s]  ← merged into chunk 3 if < 1/3 of chunkDuration
```

It walks forward by `chunkDuration - overlap` steps with no awareness of speech, silence, or speaker boundaries. The only "smart" behavior is merging a short tail chunk (< 1/3 of `chunkDuration`) into the previous one (lines 100-110). The overlap region is what compensates for mid-sentence cuts — during finalization, duplicated subtitle entries in the overlap zone are deduplicated by keeping the later chunk's version.

ASSISTANT: ### What is ref.srt?

The **reference SRT** is an existing English subtitle file for the same video — typically human-authored or from a prior transcription. It serves two purposes:

1. **Context for the LLM** — the matching SRT chunk is injected into the translation prompt alongside the audio transcript, so the LLM can cross-reference line IDs, text, and timing
2. **Ground truth for validation** — the validator compares the LLM's output against it (subtitle count, ID coverage, timing consistency)

It's optional. Without it, you lose validation checks 2-4 and must use `--use-response-timings` to get timing from the LLM instead.

### How validation works

No LLM involved — it's pure arithmetic against the reference SRT chunk:

| Check | Formula | Threshold | Severity |
|-------|---------|-----------|----------|
| **Parsing error rate** | `parser_errors / ref_count` | > 10% → fail | error |
| **Count match** | `parsed_count / ref_count` | < 80% → fail | error |
| **ID coverage** | `1 - missing_ids / ref_count` | < 80% → fail | error |
| **Timing consistency** | Per-entry: `abs(startDiff)` or `abs(durationDiff)` > 3.0s | > 25% mismatched → fail | error* |

*Timing failures are downgraded to warnings on the last chunk's final attempt.

### Is merging done with an LLM?

No. Merging is entirely deterministic code in `finalizer/index.ts`. It:
- Groups parsed entries by ID, keeps the one from the **highest-numbered chunk** (overlap resolution)
- Looks up original SRT for timing (or uses LLM timing if `--use-response-timings`)
- Runs iterative overlap repair and duration clamping

ASSISTANT: Because of how the overlapping time windows work. Consider:

```
Chunk 2: [1080s ————————————— 2280s]
Chunk 3:          [2160s ————————————— 3360s]
                  ◄─────►
                  overlap zone
                  2160s-2280s
```

A subtitle at, say, 2200s appears in both chunks. But in each chunk it sits at a **different position relative to surrounding context**:

- **Chunk 2**: subtitle 2200s is near the **tail end** — the LLM has been processing 20 minutes of audio and this line is at the very edge, where context is thinning out and the model is wrapping up
- **Chunk 3**: subtitle 2200s is near the **beginning** — the LLM has fresh context ahead of it, and the line benefits from what comes after it for translation quality

The assumption is that a subtitle near the **start** of a chunk gets better translation than one near the **end**, because the LLM has forward context (the conversation that follows) to disambiguate meaning, tone, and speaker intent. So "keep the higher chunk" means "keep the version where this line had more future context available."

### There are two completely separate timing paths

**Path A: Reference SRT timings (default)**

The ref.srt already has precise `HH:MM:SS,mmm --> HH:MM:SS,mmm` timings from whoever originally authored it. The translation prompt on line 4 explicitly says:

> *"Use the EXACT timing from the reference English subtitle"*

The LLM copies the timing into `<original_timing>` tags verbatim. During finalization, if `--use-response-timings` is **not** set (the default), these LLM-echoed timings are **ignored entirely** — the finalizer looks up the original ref.srt entry by ID and uses its timing directly (`finalizer/index.ts:164-167`). So the audio transcription timing doesn't affect subtitle display at all in this path.

**Path B: LLM response timings (`--use-response-timings`)**

If there's no ref.srt, or you pass `--use-response-timings`, the system uses the timings the LLM wrote in `<original_timing>`. These originate from:

1. Gemini transcribes audio → outputs relative `mm:ss - mm:ss` per line
2. `adjustTranscriptTimestamps()` adds the chunk's start offset → converts to absolute `HH:MM:SS,mmm --> HH:MM:SS,mmm`
3. This adjusted transcript is fed to the translation LLM
4. The translation LLM is supposed to echo/adapt those timings into `<original_timing>`
5. Parser extracts them → finalizer uses them

### Is there any smoothing?

No intelligent smoothing or alignment. The only post-processing is mechanical:

- **Overlap repair**: if subtitle N ends after subtitle N+1 starts, shorten N's end to `N+1.start - 50ms` (iterative, up to 10 passes)
- **Duration clamping**: force all durations into [0.5s, 7.0s]

There's no speech-activity detection, no silence-based boundary snapping, no forced alignment (like Whisper's word-level timestamps), and no cross-referencing of audio energy with subtitle boundaries. The Gemini transcription prompt even says *"Don't worry about getting the timestamps correct"* (`gemini_transcriber.ts:95`). The system fundamentally trusts the ref.srt for timing accuracy and treats audio transcription as a **content source only**, not a timing source.
